# Error Handling

The WebLinq API uses conventional HTTP response codes to indicate the success or failure of an API request. In general:

- `2xx` indicates success
- `4xx` indicates an error that failed given the information provided
- `5xx` indicates an error with our servers

<Note>All error responses include a detailed error message and a link to relevant documentation.</Note>

## Error Response Format

All API errors follow a standard format:

<CodeGroup>
```json Error Response Format
{
  "error": {
    "code": "error_code",
    "message": "A human-readable message explaining the error",
    "docs_url": "/api-reference/errors#error_code",
    "details": {  // Optional additional error details
      "field": "specific_field",
      "reason": "specific reason"
    }
  }
}
```
</CodeGroup>

## Common Error Codes

### Authentication Errors (401, 403)

<AccordionGroup>
<Accordion title="invalid_auth" icon="key">
  The API key provided is invalid or has been revoked.

```json
{
  "error": {
    "code": "invalid_auth",
    "message": "Invalid API key provided",
    "docs_url": "/api-reference/errors#invalid_auth"
  }
}
```

**Common causes:**

- API key is incorrect or malformed
- API key has been revoked
- API key is not included in the request

**Solution:**

<ol>
  <li>Check that you're using the correct API key</li>
  <li>Verify the key in your [dashboard](https://weblinq.dev/dashboard/api-keys)</li>
  <li>Ensure the Authorization header is properly formatted</li>
</ol>
   </Accordion>

<Accordion title="permission_denied" icon="lock">
  The API key doesn't have permission to perform the requested operation.

```json
{
  "error": {
    "code": "permission_denied",
    "message": "Your API key doesn't have permission to access this resource",
    "docs_url": "/api-reference/errors#permission_denied"
  }
}
```

**Common causes:**

- Using a read-only API key for write operations
- Attempting to access a premium feature with a standard plan
- Accessing resources owned by another user

**Solution:**

<ol>
  <li>Check your API key permissions in the dashboard</li>
  <li>Upgrade your plan if needed</li>
  <li>Use the correct API key for the operation</li>
</ol>
   </Accordion>
</AccordionGroup>

## Best Practices

### Implementing Retries

We recommend implementing exponential backoff when retrying failed requests:

<CodeGroup>
```javascript Node.js
async function makeRequest(url, maxRetries = 3) {
  for (let i = 0; i < maxRetries; i++) {
    try {
      const response = await fetch(url, {
        headers: { 'Authorization': 'Bearer YOUR_API_KEY' }
      });
      
      if (response.status === 429) {
        // Rate limited - get retry time from headers
        const retryAfter = response.headers.get('Retry-After');
        await sleep(retryAfter * 1000);
        continue;
      }
      
      return await response.json();
    } catch (error) {
      if (i === maxRetries - 1) throw error;
      // Exponential backoff
      await sleep(Math.pow(2, i) * 1000);
    }
  }
}

function sleep(ms) {
return new Promise(resolve => setTimeout(resolve, ms));
}

````

```python Python
import time
import requests
from requests.exceptions import RequestException

def make_request(url, max_retries=3):
    for i in range(max_retries):
        try:
            response = requests.get(
                url,
                headers={'Authorization': 'Bearer YOUR_API_KEY'}
            )

            if response.status_code == 429:
                # Rate limited - get retry time from headers
                retry_after = int(response.headers.get('Retry-After', 5))
                time.sleep(retry_after)
                continue

            return response.json()
        except RequestException as e:
            if i == max_retries - 1:
                raise e
            # Exponential backoff
            time.sleep(2 ** i)
````

</CodeGroup>

### Error Monitoring

Set up proper error monitoring in production:

<CodeGroup>
```javascript Node.js
function logApiError(error, context) {
  // Add request ID for support
  const requestId = error.headers?.get('X-Request-ID');
  
  console.error({
    code: error.error.code,
    message: error.error.message,
    requestId,
    context,
    timestamp: new Date().toISOString()
  });
  
  // Send to your error monitoring service
  // e.g., Sentry, Datadog, etc.
}
```
