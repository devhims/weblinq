# API Overview

The WebLinq API provides powerful endpoints for web scraping, content extraction, and data transformation. Our API follows REST principles and uses standard HTTP methods.

<Note>
  Looking for examples? Check out our [quickstart guide](/quickstart) or jump straight to the [endpoint
  reference](#available-endpoints).
</Note>

## Making your first request

<CodeGroup>
```bash cURL
# Test the API with a simple status check
curl -X GET "https://api.weblinq.dev/v1/system/status" \
  -H "Authorization: Bearer YOUR_API_KEY"
```

```javascript Node.js
// Test the API with a simple status check
const response = await fetch('https://api.weblinq.dev/v1/system/status', {
  headers: {
    Authorization: 'Bearer YOUR_API_KEY',
  },
});

const data = await response.json();
console.log('API Status:', data.status);
```

```python Python
# Test the API with a simple status check
import requests

response = requests.get(
    'https://api.weblinq.dev/v1/system/status',
    headers={'Authorization': 'Bearer YOUR_API_KEY'}
)

data = response.json()
print('API Status:', data['status'])
```

</CodeGroup>

## Base URL

All API requests should be made to:

```bash
https://api.weblinq.dev
```

<Warning>Always use HTTPS for API requests. HTTP requests will be rejected.</Warning>

## Authentication

All endpoints require authentication using a Bearer token. Include your API key in the Authorization header:

<CodeGroup>
```bash cURL
curl -H "Authorization: Bearer YOUR_API_KEY" https://api.weblinq.dev/v1/...
```

```javascript Node.js
const headers = {
  Authorization: 'Bearer YOUR_API_KEY',
};
```

```python Python
headers = {
    'Authorization': 'Bearer YOUR_API_KEY'
}
```

</CodeGroup>

<Note>Get your API key from the [dashboard](https://weblinq.dev/dashboard/api-keys).</Note>

<Warning>Never expose your API key in client-side code. Always make API calls from your backend.</Warning>

## Rate Limiting

The API implements rate limiting to ensure fair usage:

<CardGroup cols={2}>
  <Card title="Standard Plan" icon="gauge-simple">
    - 100 requests per minute - Suitable for most use cases
  </Card>
  <Card title="Premium Plan" icon="gauge-high">
    - 1000 requests per minute - Priority support
  </Card>
</CardGroup>

Rate limit information is included in the response headers:

<ResponseField name="X-RateLimit-Limit" type="number">
  Total requests allowed per window
</ResponseField>

<ResponseField name="X-RateLimit-Remaining" type="number">
  Remaining requests in the current window
</ResponseField>

<ResponseField name="X-RateLimit-Reset" type="number">
  Time when the rate limit resets (Unix timestamp)
</ResponseField>

<Note>Need higher limits? Contact us about our enterprise plans.</Note>

## Error Handling

All API errors follow a standard format. See our [Error Handling Guide](/api-reference/errors) for details about error codes and troubleshooting.

<CodeGroup>
```json 400 Bad Request
{
  "error": {
    "code": "invalid_parameter",
    "message": "The 'url' parameter is required",
    "docs_url": "/api-reference/errors#invalid_parameter"
  }
}
```

```json 401 Unauthorized
{
  "error": {
    "code": "invalid_auth",
    "message": "Invalid API key provided",
    "docs_url": "/api-reference/errors#invalid_auth"
  }
}
```

```json 429 Too Many Requests
{
  "error": {
    "code": "rate_limit_exceeded",
    "message": "Rate limit exceeded. Please try again in 60 seconds",
    "docs_url": "/api-reference/errors#rate_limit_exceeded"
  }
}
```

</CodeGroup>

<Tip>Always implement proper error handling and retries in your integration.</Tip>
